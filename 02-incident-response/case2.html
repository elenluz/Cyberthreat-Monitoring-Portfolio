Incident Triage, Ticketing & Reporting

● Documented Incident Response osTicket Incident Report Writing

Overview

This case covers the part of SOC work that happens after detection: deciding what matters, tracking it through to resolution, and writing it up clearly enough that another analyst could pick up the case cold. Findings from the CASE-001 Splunk dashboard work fed directly into this workflow.

Workflow

Alert / Finding → Ticket Logged → Triage → Investigation → Report → Close
  1. Log it — every alert or finding worth tracking was entered into osTicket as a ticket, rather than left as a one-off search result.
  2. Triage it — each ticket was scored for priority before deep investigation, using the criteria below.
  3. Investigate — pulled supporting evidence (Splunk searches, packet captures, honeypot logs) to confirm or rule out the ticket.
  4. Report it — higher-priority tickets were written up using the incident report template.
  5. Close it — ticket updated with final status and a link to the report.

🎫 TICKET — TKT-003

SQL Injection Attack on E-Commerce Web Application

Date & Time
August 11, 2017, 2:40 PM Kill Chain Phase
Exploitation MITRE ATT&CK
T1190 · T1059.001 Affected Host
172.31.38.181 Attacker IP
45.77.65.211

📋 Scenario

The web application firewall and Suricata IDS both logged alerts against the Frothly web server. External IP 45.77.65.211 was observed sending crafted HTTP requests containing SQL injection strings (UNION SELECT) targeting the web application. Two requests returned unusually large HTTP responses, suggesting the backend database may have returned unauthorized records and potential data exfiltration.
⬇ Ticket → Report ⬇
📄 INCIDENT REPORT — triage-report_severity_CRITICAL.md
Executive Summary

Summary: The incident was detected when both the WAF and Suricata IDS triggered alerts regarding anomalous traffic directed at the internal Frothly web server. Log evidence confirmed a total of 778 injection attempts. While defensive systems successfully blocked 4 requests, the vast majority (774 requests) returned an HTTP 200 OK. At least two of these successful requests generated unusually large response bodies exceeding 90 KB, confirming data exfiltration.

IOCs Identified:

IP: 45.77.65.211 Host: 172.31.38.181 SQL Injection: UNION SELECT

Containment: Block attacker IP at perimeter firewall and WAF. Suspend affected web application. Rotate database service account credentials.

Eradication: Identify vulnerable input parameter and remediate SQL injection flaw using parameterized queries. Harden WAF ruleset.

Recovery: Perform forensic audit of web and database logs. Notify stakeholders if sensitive data confirmed exfiltrated. Restore from known-good backup.

Lessons Learned: WAF must be transitioned to active blocking mode for high-confidence SQLi signatures. Refine alerting thresholds for anomalous response sizes to detect data-returning SQLi attempts earlier.

Suspicious Phishing Email With Malicious Attachment

Date & Time
August 24, 2017, 3:27 AM Kill Chain Phase
Delivery MITRE ATT&CK
T1566.001 · T1203 Affected Host
wrk-btun (billy.tun) Sender
jsmith@urinalysis.com

📋 Scenario

A Frothly employee reported receiving a suspicious email containing a password-protected attachment (invoice.zip). The mail gateway flagged the message and SPF verification for the sender failed; the sender domain appears recently registered. The user opened the document, triggering the execution of malicious code via MS Word.
⬇ Ticket → Report ⬇
📄 INCIDENT REPORT — triage-report_severity_HIGH.md
Executive Summary

Summary: A spearphishing email with a password-protected attachment was delivered to the environment. The user opened the document, triggering the execution of malicious code via MS Word. Indicators of suspicious behavior were observed on the affected host and in email headers.

IOCs Identified:

File: invoice.zip File: invoice.doc (Trojan) Sender: jsmith@urinalysis.com

Containment: Immediately isolate affected workstation. Block sender domain at email gateway.

Eradication: Scan endpoint to remove malicious file. Disable compromised user account billy.tun and reset password.

Recovery: Re-image workstation from secure baseline. Monitor environment for anomalous traffic patterns.

Lessons Learned: Initial delivery bypassed AV/Mail Gateway due to password protection. Requires improved inspection of password-protected attachments at mail gateways and behavioral monitoring of spawned processes from MS Word.
P3 – Medium Suspicious but plausible benign explanation ▶ TKT-009 · Brute-Force RDP Attack
🎫 TICKET — TKT-009

Brute-Force Authentication Attack on RDP — Successful Logon

Date & Time
June 29, 2017, 10:51 AM Kill Chain Phase
Exploitation MITRE ATT&CK
T1110.001 · T1078 Affected Host
10.0.1.100 Attacker IP
71.39.18.121

📋 Scenario

Windows Security logs on a Frothly Domain Controller recorded hundreds of consecutive failed logon attempts against the Administrator account from a single external IP address late at night. The attempts were followed by a successful logon from the same IP using Remote Desktop Protocol. A successful RDP logon to a Domain Controller by an external IP is a critical severity incident requiring immediate escalation.
⬇ Ticket → Report ⬇
📄 INCIDENT REPORT — triage-report_severity_MEDIUM.md
Executive Summary

Summary: A sophisticated, multi-stage attack was detected against the Frothly Domain Controller. The incident began with the compromise of host 10.0.1.100 (Patient Zero), which was used to perform automated lateral movement and brute-force reconnaissance against the internal network. This culminated in a successful interactive compromise of the Administrator account by an external threat actor.

IOCs Identified:

IP: 71.39.18.121 Host: 10.0.1.100 Account: Administrator Lateral Movement: 10.0.2.109, 10.0.1.101, 10.0.2.107

Containment: Immediately isolate 10.0.1.100. Block 71.39.18.121 at perimeter firewall. Disable compromised Administrator account.

Eradication: Mandatory password resets for Administrator account and all accounts that logged into 10.0.1.100. Execute KRBTGT password reset (twice). Audit for newly created accounts and persistence mechanisms.

Recovery: Re-image affected hosts. Strictly limit RDP access using VPN + MFA. Continue enhanced monitoring for anomalous logon activity.

Lessons Learned: Environment lacked automated alerting for high-volume failed logon attempts. Domain Controller was directly exposed to the internet via RDP. Need threshold-based alerts and Zero Trust approach for administrative RDP access.
P4 – Low Informational / noise, logged for trend tracking ▶ TKT-010 · Drive-By Download
🎫 TICKET — TKT-010

Drive-By Download via Malicious Ad Network Redirect

Date & Time
August 15, 2017, 11:35 AM Kill Chain Phase
Delivery MITRE ATT&CK
T1189 · T1204.001 Affected Host
10.0.2.109 Attacker IP
45.77.65.211

📋 Scenario

Proxy and HTTP stream logs show a Frothly workstation visiting a legitimate website that triggered a redirect chain through multiple hops ending at a suspicious external domain. A JavaScript file was downloaded followed by a Windows executable. No antivirus alert was generated at the time, highlighting a detection gap. The executable filename mimics a legitimate system service.
⬇ Ticket → Report ⬇
📄 INCIDENT REPORT — triage-report_severity_LOW.md
Executive Summary

Summary: A Frothly workstation (10.0.2.109) was compromised via a drive-by browser exploit. The workstation visited www.brewertalk.com which redirected through a malicious chain that delivered an XSS payload. The payload exfiltrated the user's authentication/session cookie to the attacker-controlled server at 45.77.65.211. The attacker hijacked the authenticated session and created a rogue account named klagerfield.

IOCs Identified:

IP: 45.77.65.211 Domain: www.brewertalk.com Account: klagerfield

Containment: Block traffic to/from 45.77.65.211 at perimeter firewall and via DNS filtering. Block www.brewertalk.com until confirmed clean.

Eradication: Disable unauthorized account klagerfield and invalidate all active sessions. Revoke elevated privileges and remove artifacts. Scan affected host for browser-based or fileless persistence.

Recovery: Remediate XSS vulnerability with proper input validation and output encoding. Apply Content Security Policy. Re-image compromised workstation if needed. Rotate affected user credentials.

Lessons Learned: Lack of browser/endpoint telemetry for suspicious script execution and reliance on signature-based AV allowed the XSS-driven session hijack to proceed undetected. Implement DNS filtering, browser isolation, and improved session token protections.
